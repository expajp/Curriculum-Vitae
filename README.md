# 自己PR
「**美しい仕組みを作る**」ことに興味のあるエンジニア/エンジニアリングマネージャーです

「美しい」とは、シンプルで分かりやすく、無駄なく問題を解決していることを指します

## エンジニアとして
### HTTPプロトコルの美しさから、Web開発が好き
* たった8つの命令、「リクエストを送るとレスポンスが返る」だけというシンプルさ、オープンプラットフォームであることによる間口の広さ
* まさに「美しい仕組み」です

### 情報収集を怠らず、常に勉強し続ける習慣
* [情報収集について書いたブログ記事](http://expajp-tech.hatenablog.com/entry/2018/12/01/013953)
* ほぼ毎日コードを書いており、「業務外の主なプロダクト」に書いたものの製作以外に[Project Euler](https://projecteuler.net/)の100問解答も達成しています
* 勉強会には頻繁に参加していますが、自分で発表する形式と講義を聞く形式のものに絞っています

### 「ソフトウェア設計」スキルに重きを置いている
* ソフトウェアは設計で8割決まると思っています
  * 設計がなぜ存在するかといえば、主に保守性のためです
  * しかし、設計が十分に検討されていることは、解決しようとする問題について十分に検討されていることを裏付けています
  * そのため、良い設計がされているソフトウェアは使いやすいことが多い、という考えを持っています
* オブジェクト指向設計実践ガイドを読んでそれに合わせてbrainf*ck処理系を実装しました
  * [expajp/brainfxck](https://github.com/expajp/brainfxck)
* エリック・エヴァンスのドメイン駆動設計の読書会に参加しています
  * [3章担当時の資料](https://hackmd.io/@AyCdH1KqQqS0bweuWIQE3A/BkpcLH6RV)

## エンジニアリングマネージャーとして
###  組織がより良くなるための仕組みを考えるのが得意
* 「より良いエンジニアリング組織」とは、メンバー個々の生産性をより大きく、不満をより小さくし続けていける組織を指します
* 「組織がより良くなるための仕組み」とは、主に以下の3つに分かれます
  * つまらない定常タスクの自動化
  * 定期的なふりかえり
  * 自由に不満が言えるような環境づくり

### 仕組みを徐々にインストールしていくことが得意
* 仕組みを組織にインストールするには、以下が重要だと思っています
  * メリットとデメリットを包み隠さず説明し、デメリットへの対策も説明する
  * 新しいSaaSなど、触れる性質のものがあれば実際に触ってもらってメリットを実感してもらう
  * それでも不満に思う人がいれば、個別に聞き取りを行って解消に努める
